# 개미지표 (Ant-Index) 구현 계획

## Context
한국 주식 커뮤니티의 감성을 분석하여 공포탐욕지수와 유사한 시장 심리 지표를 만드는 1인 사이드 프로젝트.
커뮤니티 비속어 기반 지표(ㅅㅂ지수)와 환희 키워드 기반 지표(가즈아지수)를 핵심으로, 시장 데이터 기반 지표를 점진적으로 확장.

## 기술 스택
- **크롤러**: Python (requests + BeautifulSoup / Playwright) — cron 배치 스크립트
- **백엔드**: Nest.js (지표 계산 + API)
- **프론트엔드**: Next.js (대시보드)
- **DB**: PostgreSQL
- **ORM**: Nest.js — Drizzle ORM (SQL-like 경량 ORM) / Python — SQLAlchemy Core (SQL 빌더, ORM 레이어 미사용)
- **오케스트레이션**: Docker Compose
- **배포**: 온프레미스 서버 (라즈베리파이 8GB 또는 미니PC)

## 데이터 소스 (MVP에서 둘 다 동시 진행)
- **네이버증권 종목토론실**: SSR, requests + BeautifulSoup으로 크롤링
- **토스증권 커뮤니티**: SPA, Playwright로 크롤링

## 데이터 흐름

```
Python 크롤러 (cron) → 댓글 수집 + 비속어 분석 → PostgreSQL에 저장
                                                         ↓
Next.js 대시보드 ← REST API ← Nest.js (DB 읽어서 지표 계산)
```

### DB 접근 아키텍처

- **Drizzle ORM** (Nest.js): 스키마 정의의 source of truth + 마이그레이션 관리 + 지표 계산/API 읽기
- **SQLAlchemy Core** (Python 크롤러): 같은 DB에 직접 쓰기 (INSERT 위주) + 중복 체크용 SELECT
- 두 레이어가 같은 PostgreSQL 테이블을 바라보되, DDL은 Drizzle이 관리
- SQLAlchemy는 ORM 레이어 없이 Core만 사용 — SQL 구조가 코드에 그대로 드러남

```
# Python (SQLAlchemy Core) — INSERT 예시
stmt = insert(comments).values(
    stock_id=1, content="텍스트", profanity_score=0.8
)
engine.execute(stmt)

# Nest.js (Drizzle) — SELECT 예시
db.select().from(comments).where(
    eq(comments.stockId, 1)
).orderBy(desc(comments.crawledAt))
```

## 핵심 설계

### 1. 커뮤니티 감성 탐지

#### ㅅㅂ지수 (공포/분노) — 비속어 탐지

| Tier | 유형 | 예시 | 가중치 |
|------|------|------|--------|
| 1 | 직접적 비속어 | 씨발, 개새끼, 병신 | 1.0 |
| 2 | 초성 비속어 | ㅅㅂ, ㅂㅅ, ㅈㄴ | 0.8 |
| 3 | 변형 비속어 | 시1발, 씨빨, ㅅ1ㅂ | 0.7 |

#### 가즈아지수 (환희/탐욕) — 긍정·환희 키워드 탐지

| Tier | 유형 | 예시 | 가중치 |
|------|------|------|--------|
| 1 | 직접적 환희 표현 | 가즈아, 떡상, 로켓, 존버 승리 | 1.0 |
| 2 | 축약/변형 | ㄱㅈㅇ, 갈께요, 떡상각 | 0.8 |
| 3 | 긍정 감성어 | 매수 찬스, 저점, 물타기 성공 | 0.5 |

#### 좋아요 가중치

좋아요가 많은 댓글은 커뮤니티 공감을 받은 의견이므로 더 높은 가중치 부여:
```
댓글 가중치 = 1 + log10(좋아요 수 + 1)
```
| 좋아요 | 가중치 |
|--------|--------|
| 0 | 1.0 |
| 10 | 2.0 |
| 100 | 3.0 |
| 1000 | 4.0 |

지표 계산 시 `키워드_가중치 × 댓글_좋아요_가중치`로 최종 스코어 산출.

**공통 구현**:
- Python `re` 모듈로 정규식 매칭 (별도 라이브러리 불필요)
- `KoreanCursewordRegex` 참고하여 비속어 정규식 패턴
- 사전은 JSON/Python dict로 인메모리 매칭 (DB 아님)
- 같은 크롤링 파이프라인에서 키워드 사전만 다르게 적용

### 2. 지표 계산

#### ㅅㅂ지수 (0~100, 높을수록 분노)
```
raw_score = (가중_비속어_수 / 전체_댓글_수) × 1000
normalized = ((raw - min_30d) / (max_30d - min_30d)) × 100  [0~100 클램프]
```

| 범위 | 레이블 |
|------|--------|
| 0-20 | 극도의 평온 |
| 20-40 | 평온 |
| 40-60 | 보통 |
| 60-80 | 불안 |
| 80-100 | 극도의 분노 |

#### 가즈아지수 (0~100, 높을수록 환희)
```
raw_score = (가중_환희_키워드_수 / 전체_댓글_수) × 1000
normalized = ((raw - min_30d) / (max_30d - min_30d)) × 100  [0~100 클램프]
```

| 범위 | 레이블 |
|------|--------|
| 0-20 | 침체 |
| 20-40 | 조용 |
| 40-60 | 보통 |
| 60-80 | 흥분 |
| 80-100 | 극도의 환희 |

#### 댓글 활성도 (0~100, 높을수록 활발)
```
raw_count = 최근 시간 댓글 수
normalized = ((raw - min_30d) / (max_30d - min_30d)) × 100  [0~100 클램프]
```
- 정규화 활성도 (0~100): 평소 대비 얼마나 뜨거운지 → `index_snapshots`에 ACTIVITY로 저장
- 절대 댓글 수: 종목 간 화력 비교 → `index_snapshots`의 `totalPosts` 컬럼 활용
- 국장/미장 모두 동일 방식 적용 (개미 거래 비율은 국장만 pykrx로 측정 가능, 미장은 공개 데이터 없음)

| 범위 | 레이블 |
|------|--------|
| 0-20 | 한산 |
| 20-40 | 조용 |
| 40-60 | 보통 |
| 60-80 | 활발 |
| 80-100 | 폭발 |

**공통**:
- **시간 단위**: 시간별 스냅샷 저장, 일별 롤업, 7일 이동평균
- **최소 댓글 기준**: 일 10개 미만 종목은 제외 또는 저신뢰 표시
- **부트스트랩**: 크롤러를 먼저 가동하여 30일간 데이터 축적 (지표 비공개) → 30일 후 지표 공개. 프론트 개발 기간과 병행하면 자연스럽게 해결

### 4. 크롤링 전략 (Python)

**네이버증권 (requests + BeautifulSoup)**:
- 종목토론실: `https://finance.naver.com/item/board.nhn?code={종목코드}` — 댓글 수집
- 종목메인: `https://finance.naver.com/item/main.nhn?code={종목코드}` — 기본 정보(현재가, 시총, PER 등) 함께 수집
- SSR이라 HTML 직접 파싱 가능
- GitHub 오픈소스 크롤러 다수 참고 가능

**토스증권 (Playwright)**:
- URL: `https://www.tossinvest.com/stocks/A{종목코드}/community`
- React SPA → headless 브라우저 필수
- Playwright가 Puppeteer보다 안정적이고 Python 생태계에 잘 맞음
- 브라우저 인스턴스 재사용으로 메모리 절약

**공통**:
- User-Agent 로테이션, 3-5초 간격
- 지수 백오프 (429/503 에러 시)
- abstract `BaseSource` 클래스로 인터페이스 통일
- cron으로 스케줄링 (OS 레벨), Python 스크립트는 실행 후 종료

### 5. 크롤링 스케줄

- **장중 (평일 09:00-16:00 KST)**: 30분마다
- **장외**: 2시간마다
- **지표 계산**: 매시간 (시간별 스냅샷) + 매일 00:05 (일별 롤업)

| **UI/레이아웃** | 깔끔, 카드, 여백, 절제 | Phantom, 토스, Robinhood |
| **그래픽/브랜딩** | 투박, 비정형, 거친 질감, 강렬한 대비 | 머니그라피, 배달의민족 |

- UI: 카드 기반, 둥근 모서리, 충분한 여백 (토스/Phantom)
- UI: 숫자가 화면 주인공 — 지표 수치를 크고 임팩트 있게 (Robinhood)
- 그래픽: 볼드하고 불규칙한 타이포 — 클레이/픽셀/손글씨 질감 (머니그라피)
- 그래픽: 블랙 베이스 + 오렌지/레드 강렬한 포인트 (머니그라피)
- 갭의 매력: 절제된 UI 안에서 로고·캐릭터·OG이미지·공유카드에서 날것의 개성이 터짐

**컬러 시스템**:
- 브랜드 포인트: 블랙 + 오렌지/레드 (머니그라피 톤)

## 온프레미스 배포 (Docker Compose)

- **하드웨어**: 라즈베리파이 8GB 또는 미니PC (추후 확정)
- **배포**: `docker compose up -d` 한 줄로 전체 서비스 구동
- **서비스 구성**: db(PostgreSQL) + crawler(Python) + server(Nest.js) + web(Next.js)
- **크롤러**: 컨테이너 내부 cron 또는 Docker restart policy로 주기 실행
- **Swap**: 2GB swap 설정 권장 (Playwright 피크 메모리 대비)
- **확장**: 트래픽 증가 시 Docker Compose → Kubernetes 마이그레이션 가능 (Dockerfile 재사용)

### Phase 4: 배포 + 고도화
- 각 서비스 Dockerfile 작성
- docker-compose.yml 구성 (db, crawler, server, web)
- 온프레미스 서버에 Docker 설치 → `docker compose up -d`
- 모니터링/로깅

## 검증 방법

1. **비속어 탐지**: 유닛 테스트로 Tier 1-3 패턴 검증 (정상 문장 false positive 확인 포함)
2. **크롤링**: 네이버/토스 각 1개 종목 크롤링 → 댓글 파싱 정상 확인
3. **지표 계산**: 시드 데이터로 raw_score → normalized 계산 결과 검증
4. **E2E**: 크롤링 → 분석 → 지표 계산 → API 응답 → 대시보드 렌더링 전체 파이프라인
